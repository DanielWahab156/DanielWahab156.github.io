Artemis Setup

The purpose of this lab was to get students situated with the arduino ide and the Sparkfun Artemis board. Most of the lab involved getting familiar with the arduino ide and uploading sample code to the artemis.

  • Parts required: 1 x SparkFun RedBoard Artemis Nano, 1 x USB C-to-C or A-to-C cable

Prelab

The prelab instructions guide on how to setup the arduino ide

Task 1: LED Blink

This first task involved uploading example code that would cause the LED to flash once uploaded to the artemis

Task 2: Serial

This second task made sure that the serial ports of the board were working properly so that we can send and receive signals from these ports.

Task 3: Analog read

This task asked us to test the sensors by uploading the example code and then blowing onto the board. This would cause changes in the reading in the activity monitor.

Task 4: Microphone

Similar to the last part, this part asked us to upload the example code to the board and speak into the board at different volumes. This tests the microphone on the board, and the expected output is that there will be changes in readings.

Bluetooth

The goal of this lab was to establish a bluetooth connection with the artemis and our own computers. Another goal of this lab was to establish more familiarity with the arduino_ble library which can be accessed here: https://www.arduino.cc/reference/en/libraries/arduinoble/

  • Parts required: 1 x SparkFun RedBoard Artemis Nano, 1 x USB C-to-C or A-to-C cable (same as the first part of the lab)

Prelab

The goal of this prelab was to help us get familiar with the arduino_ble codebase and also install the python environment that most of the lab would take place in. The code used to create the working jupyter notebook is below:

python3 -m pip install --user virtualenv
python3 -m venv FastRobots_ble

The following code was how to activate/deactivate the jupyter notebook in the working directory.

source FastRobots_ble/bin/activate

deactivate

After this, the next goal was to establish the bluetooth connection. This was done by running the ble_arduino.ino file, which returns that MAC address of the board below.

Then students were given the sample code below to generate a sample uuid. The UUID and the MAC address were both entered in the yaml file which allowed us to establish a bluetooth connection with the arduino.

Task 1: ECHO

This first task involved creating a new command, ECHO, which sends a string to the artemis rx_characteristic_string which is then recieved back to the artemis. This lab is completed by writing code on arduino side, appending it to an estring value, and appending it to the tx_characteristic_string.

Task 2: Get_Time_Millis

This second task asked us to create a new command again, but this time the goal of the command was to print out the current time. I used a similar process as last time (appending to tx_characteristic_string) to send the time to the board so it can be displayed in python. This time, I just used the millis() function to access the time.

Task 3: Notification Handler

This third task consisted of creating a notification handler which is done with ble.start_notify. The first argument is the RX_String which is what we want to be notified of when changes are occuring. The next argument if the function for a notification handler which we created.

Task 5: Sending Time Data

For this part, the goal was to update the time in an array in the loop function and read through the array in a new created command. This was done by declaring a new array in loop and updating it per millisecond. After, the reading through was done in a seperate command.

Task 7: Comparing methods

In terms of efficiency, both methods have there upsides and downsides. The first method is better for constant updates while the second method is better when we need to check when data is sent. In terms of how quickly data is sent, since data is 16 bits, the microcontroller runs at 150 Hz, and we use 3072000 bits of data we can store almost 1300 bits/sec before running out of data.

Lab 3: Time of Flight Sensors

The main objective for this lab is to equip multiple ToF sensors onto the robot so it can get more sample readings at a faster rate. This allows for the robot to get more accurate and much faster data readings. Using 2 Time of Flight sensors allow us to gain better coverage, improving accuracy and speed.

  • Parts required: Sparkfun Artemis Nano, 1x USB cable, 2x 4m ToF sensor, 1x QWIIC Breakout Board, 2x QWIIC connectors, 1x ruler

Prelab

The goal of the prelab was to have an idea of the wiring schematic for the project. The prelab served to help you plan out how you were to solder the two ToF sensors onto the QWIIC connect cables and to the artemis. My sample schematic can be seen below. The wires were connected to the ports they were meant for on the ToF sensors. The single wire between one ToF sensor and the artemis was to shut off the board to give both sensors different I2C addresses since the default I2C address is 0x52.

Task 1: Single ToF Sensor

This first task involved soldering on one of the ToF sensors to the QWIIC connect cables and plugging them into the board. This gave us distance readings whenever moving around the TOF sensor. A video can be shown below with the readings recieved from the TOF sensor. The picture below also shows the wiring used for the single ToF sensor.

One part of the lab involves checking the I2C address by uploading the exaple of Wire-to-I2C onto the board and reading the address that is printed out. When analyzing the output, it was observed that the address printed out was 0x29. This was not the expected outcome because in the manual, the default I2C address was supposed to be 0x52.

Task 2: Determining Distance Mode

For this lab, we had available two different distance modes to choose from. Either short distance or long. The long distance is better for more accurate readings from further away but take longer to transmit. The short distance is better for much quicker readings but may not be as accurate. I went for the long distance mode as it helps us gain better readings from further distances which is good for fast robots.

Task 4: Testing ToF Sensors

After determining a distance mode and figuring out how to get readings from the time of flight sensors, the next task involves testing accuracy and range. To do this, I slowed down the print rate of the time-of-flight sensors distance and held the sennsor at different distances from the computer, each time getting further. I then used the first three distance measurements and plotted them relative to the actual distance. The graph and setup can be seen below.

Task 5: Two ToF Sensors

The next task involves soldering on the second time-of-flight sensor. For this task, the second sensor needs to have an extra wire soldered to the xshut pin because we shut off the second ToF sensor to change the address so we can get the different distance measurements at the same time. The picture of the setup with the simultaneous distance measurements is below.

Lab 4: Motor Driver

The main purpose of this lab was to establish the connection between the artemis and the motors on the robot via motor drivers and connect each part of the previous 2 labs together. This lab involved lots of soldering in order to get the motors working and controlled by the artemis.

Prelab: The prelab involved diagraming the wiring for the motor drivers before actually soldering the Parts together onto the artemis. The diagram can be seen right below this. The methodology was that the Artemis was meant to send a pwm signal via pins to the input of the motor driver. Then, the motor driver would output the current needed for the motors to start running at that pwm value. The motor drivers also need to be connected to ground of the artemis and needed to connect to the positive and negative terminals of vin and gnd respectively.

Below is an image of the wiring set up for the motor drivers including the full robots circuit with everything wired together and connect by tape and the soldering of the motor drivers. Because both the 2 input and output pins should be shorted together(respectively), I soldered the back to connect in1/in2 and out1/out2.

Next was to make sure the wheels would spin for each pwm value. The pins I soldered the motor driver inputs to are pins 6,7,11,13. In the videos below, I checked the motors would work for a default pwm value I used(155). What can be seen is that each pin controls one side/direction of the wheels. Pins 11 and 13 control the same wheels but opposite spins of the wheels and the same is for pins 6 and 7. The AnalogWrite code is also posted right below this as well.

Next up was to find the lower limit pwm value for the motors to spin. In the video below, you can see that after some testing I found that pwm of 40 would cause the wheels to cease to spin.
